# SpikingRx-on-OAI 串接訓練 Debug 完整筆記

## 0. 這個聊天室的核心目標

這次聊天的主軸不是再去改 OAI dump 流程本身，而是：

**把目前已經存在的 supervision 鏈整理清楚，確認其數學意義，然後開始修「串接式訓練 pipeline + 模型」。**

最終目標仍然是：

```text
fullgrid  ->  model  ->  llr
```

而且這個 model 應該是**單一整體模型**，不是三個獨立模型。

訓練時則使用中間 supervision：

```text
fullgrid
  -> ch head      (supervise by chestext)
  -> eq/rxcomp    (supervise by eqsymb)
  -> llr head     (supervise by demapper_llr_f32)
```

## 6. 原始訓練 script 的問題

你原本的 `train_spikingrx_oai.py` 有兩個大問題：

### 6.1 LLR 只學 sign，不學值

原始版本是：

```python
y_llr_sign = (y_llr >= 0).float()
loss_llr = BCEWithLogitsLoss()(llr_pred, y_llr_sign)
```

也就是只要求：

```text
sign(llr_pred) ≈ sign(demapper_llr_f32)
```

但沒有要求：

```text
llr_pred ≈ demapper_llr_f32
```

### 結論

這太弱，後面 decoder 需要的不只是正負號，還需要可靠度大小。

### 6.2 eq loss 名義上有，實際上沒打開

你之前有一版：

```text
w_eq = 0.0
enable_eq_loss = True
```

這其實等於沒有真正訓練 eq。

## 10. 接著改 model，把 SNN「救活」

因為原始 `spikingrx_model.py` 的問題不是 pipeline 接法，而是：

- activation 太小
- threshold 太高

所以後來的修改方向是：

### 10.1 修改 `spikingrx_model.py`

主要改了這兩個核心點：

#### (1) 降低 LIF threshold

從：

```text
theta = 0.5
```

改成：

```text
theta = 0.2
```

#### (2) 在 StemConv 裡放大 activation

加上：

```python
z = z * self.input_gain
```

而 `input_gain = 5.0`

### 目的

讓 spike 不再幾乎全死。

## 14. 下個聊天室的建議 debug 順序

下面是最建議的順序，不要再亂跳。

### Step 1：先檢查 llr head 是否「scale 不夠」

現在看到：

- `eq_pred std` 已經能到 `0.8+`
- 但 `llr_pred std` 還只有 `1.7~1.8`
- `y_llr std ≈ 11.5`

### 要做的事

檢查：

- llr head 是否輸出範圍太小
- `llr_temperature=1.0` 是否需要放大
- 或 llr readout 的 hidden / pool / FC 容量不足

### 很可能的方向

先試：

- 把 `llr_temperature` 提高，例如 `2.0 / 4.0`
- 或讓 llr loss 的 value 部分權重再提高

### Step 2：直接驗證 `eq_pred` 能不能單獨推出 LLR sign

既然你已經知道真值：

```text
eqsymb / chestext -> QPSK
```

那現在要檢查：

### 做法

在 train / eval 裡另外算一個簡單 baseline：

- 用 `eq_pred` 的 real / imag sign 去對應兩個 QPSK bit sign

看它能不能比 `0.509` 好

### 目的

如果這個簡單規則都能 `>0.6`，但 llr head 還是 `0.509`，代表：

- 問題在 llr readout 本身
- 不是前面 `ch/eq` 不夠好

### Step 3：檢查 `aux_normalize=True` 是否讓 llr mapping 變難

現在 `y_ch / y_eq` 都做了 sample-wise masked normalize。

這對 `ch/eq` supervision 本身有幫助，但可能讓 llr head 難學，因為：

- `eq_pred` 學到的是 normalized `H*Y`
- llr target 卻是實際 `demapper LLR`

也就是 llr head 要同時學：

- 還原 `H / scale`
- 再學 demapper

### 可以試的方向

下個聊天室可測：

- 保持 `y_ch normalize`
- 但 `y_eq` 改成不 normalize，或改成固定 global scale，而不是每 sample z-score

這很可能是下一個關鍵點。

### Step 4：增加 phase 長度，讓 `ch/eq` 先學穩

目前 `phase1 / phase2` 可能太短：

- `phase1` 只有 5 epoch
- `phase2` 只有 7 epoch

但你看 log，`eq loss` 到 epoch 11 還在穩定下降。

### 代表

可能需要先讓：

- ch 先收斂
- eq 再收斂
- 最後再主打 llr

### 可試方向

例如：

```text
phase1: 1~8
phase2: 9~20
phase3: 21~40
```

### Step 5：必要時把 llr head 獨立加強

如果前面都確認沒問題，就要改 `LLRFusionReadout`：

### 可試方向

- 增加 hidden conv channel（32 -> 64）
- 增加一層 conv
- 增加 hidden_fc（256 -> 512）
- 調整 `pool_hw`
- 甚至直接讓 llr head 吃 `eq_pred/ch_pred` 後再做更淺顯的 demapper-style head
