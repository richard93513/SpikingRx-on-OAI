# SpikingRx-on-OAI：SPX dumps / bundling / ldpctest_spx（本聊天室版本筆記）

## 0) 目的（為什麼要做這些 dump）
最終目標是把「UE pipeline 裡 **真正餵進 LDPCdecoder 的輸入**」與「離線 ldpctest_spx 的 LDPCdecoder 輸入」做到一致，
並且把比對範圍逐步縮小：

1. 先確認：`rm_exact`（LDPCdecoder input, int8, out_len layout）**檔案不會撞檔、不會被覆蓋**，而且可穩定定位到 (frame, slot, idx, tb, rv, seg)。
2. 再確認：離線 `ldpctest_spx` 只做 **decoder**（BP/min-sum），不做 deinterleave / unmatching，避免把錯誤歸因搞混。
3. 最後用 `cmp` 做 byte-by-byte 對照：
   - `cb_payload_segXX.bin` vs `ue_c_segXX.bin`
   - `tb_payload_bytes.bin` vs `ue_tb.bin`

如果 CB/TB 都能完全一致，就表示：
- UE 端 rm_exact dump 正確
- 離線 LDPC decoder 程式正確
- 問題若仍有 BER/BLER異常，應該在「更上游：LLR/demapper/SpikingRx 產生 LLR」或 bundling/對齊流程

---

## 1) 全域識別與 idx scheme（四檔共通）
- idx scheme：`idx = frame * 100 + slot`
- 所有檔名都包含：`f%04d_s%02d_idx%06d`
- 本聊天室版本新增關鍵：**rm_exact 一定要包含 TB id (`tb%02u`)**，避免同 frame/slot 多 TB 或平行 thread 覆蓋撞檔。

## 2) 四個 dump 檔案：位置、輸出、檔名規格與重點

### 2.1 UE：`nr_dlsch_demodulation.c`
**用途**
- Dump UE 端 OFDM full-grid（供 SpikingRx 使用）
- Dump UE 端 demapper LLR（float32，供 debug/對照，不是 decoder input）

**輸出檔名（你目前截圖看到的命名）**
1) Fullgrid（binary）
- `f%04d_s%02d_fullgrid_idx%06d_rnti%05d_harq%02d.bin`

2) Demapper LLR（float32）
- `f%04d_s%02d_llr_idx%06d_rnti%05d_harq%02d_f32.bin`
  - 重點：dtype 是 `float32`，建議檔名帶 `_f32` 避免混淆

**重點/注意**
- 這份 LLR 是 demapper output（或 demod softbits），**不是 LDPCdecoder input**。
- bundler 內通常把它改名成 `demapper_llr_f32.bin` 只留 debug 用。

---

### 2.2 gNB：`nr_dlsch_coding.c`
**用途**
- Dump gNB TX bits（ground truth for BER）
- Dump gNB LDPC cfg JSON（對照參數）

**輸出檔名（你貼的那段 code 格式）**
1) gNB LDPC cfg JSON
- `f%04d_s%02d_ldpc_idx%06u_rnti%05d_pdu%03d_rv%01d_tbcrc%08x.json`

2) gNB TX bits
- `f%04d_s%02d_txbits_idx%06d_rnti%05d_pdu%03d_rv%01d_tbcrc%08x.bin`

**重點/注意**
- gNB 的 rnti 目前是十進位 `%05d`（和 UE decoding 那邊曾經用 `%04x` 不一致，會讓 regex 解析麻煩；但你說你會清 raw，就先以目前為準）
- tbcrc 用來挑 “最佳候選” 也很方便（例如 tbcrc==00000000 + pdu==0）

---

### 2.3 UE：`nr_dlsch_decoding.c`
**用途**
- Dump UE LDPC cfg JSON（包含 rv/round/dlsch/harq 等，用來決定 bundle key）
- Dump UE decoder output（用來做最終 byte-by-byte 驗證）

**UE LDPC cfg JSON（典型檔名）**
- `f%04d_s%02d_ldpc_idx%06d_rnti%04x_dlsch%02d_harq%02d_round%02d_rv%01d.json`
> 你 bundler 目前 regex 寫成「UE LDPC JSON 沒有 rnti」，但你截圖裡看到的那個檔名其實帶 rnti1234（十進位），所以**一定要統一**：
> - 要嘛：UE LDPC JSON 檔名固定不帶 rnti（bundler 也照這個）
> - 要嘛：UE LDPC JSON 檔名固定帶 rnti（bundler regex 改成有 rnti）
>（你目前 bundle 已經能產出，代表你現在 raw 檔名與 bundler regex 是配套好的；之後改檔名要同步改 regex。）

**UE decoder output dumps**
1) UE TB bytes（A/8 bytes）
- `f%04d_s%02d_ue_tb_idx%06d_rnti%04x_dlsch%02d_harq%02d_round%02d_rv%01d.bin`

2) UE CB payload per segment（594 bytes 這種）
- `f%04d_s%02d_ue_c_idx%06d_rnti%04x_dlsch%02d_harq%02d_round%02d_rv%01d_seg%02d.bin`

**重點/注意**
- 這兩個是你最後要拿來跟 ldpctest_spx 做 `cmp` 的關鍵檔：
  - `ue_c_segXX.bin`（bundle內改名後）
  - `ue_tb.bin`
- 這邊最重要是：dump 的必須是 **decoder output 後** 的 bytes，不要 dump 到 decoder input（你之前就踩過這坑）。

---

### 2.4 共用 decoder segment：`nrLDPC_coding_segment_decoder.c`
**用途**
- Dump **LDPCdecoder 的真正輸入**：int8 buffer `l[]`（out_len layout）
- 這是最關鍵的 “rm_exact gate”

**本聊天室版本改動重點（防撞檔）**
- 在 `nrLDPC_decoding_parameters_t` 增加：
  - `spx_frame, spx_slot, spx_idx`
  - `spx_tb`（= pusch_id / TB index）✅ *防撞檔核心*
  - `r`（seg index）

- 在 `nrLDPC_prepare_TB_decoding(..., int pusch_id, ...)` 內把上述 identity 填好傳進 thread pool 的 rdata

**輸出檔名（你現在這版）**
- `f%04u_s%02u_ldpc_rm_exact_idx%06u_tb%02u_rv%01u_seg%02u_outlen%05d_i8.bin`

**重點/注意**
- `outlen%05d` 的 `05` 只是**補零寬度**（顯示格式），沒有功能性問題。
- `tb%02u` 是必需的，否則多 TB / 平行 segment 會撞檔覆蓋，造成 “看起來 dump 有，但其實內容被覆蓋”的幽靈 bug。

## 3) Bundle 資料夾內容（目前版本）
每個 bundle 目錄（例如 `f0696_s01_idx069601_rnti1234_harq06_rv0/`）包含：

- `fullgrid.bin`
- `demapper_llr_f32.bin`               （float32，debug）
- `rm_exact_seg00_i8.bin`              （int8，out_len，直接餵 LDPCdecoder）
- `rm_exact_seg01_i8.bin`              （若 C=2）
- `txbits.bin`                          （gNB ground truth）
- `ue_ldpc.json`
- `ldpc_cfg.txt`                        （供 ldpctest_spx 使用）
- `ue_tb.bin`                           （UE decoder TB bytes）
- `ue_c_seg00.bin`, `ue_c_seg01.bin`    （UE decoder CB payload bytes）
- `meta_bundle.json`

**bundle 的核心定位**
- `rm_exact_*`：你要拿去餵離線 LDPCdecoder
- `ue_c_*`、`ue_tb.bin`：你要拿來跟離線結果做 cmp 的 “oracle”
- `demapper_llr_f32.bin`：保留 debug（不是本輪 cmp 的核心）

## 4) ldpctest_spx 改動（本聊天室版本）
### 4.1 改動目的
把 ldpctest_spx 的責任縮小到：
> **只做 LDPCdecoder（BP/min-sum）**  
不做 deinterleave / rate-unmatching，避免錯誤歸因混雜。

### 4.2 輸入規格（只吃 rm_exact）
- 只接受 `rm_exact` int8
- 支援兩種輸入形式：
  1) concat 檔：size = `out_len*C` bytes
  2) per-seg 檔：指定 `rm_exact_seg00_i8.bin`，程式會自動載入 seg01..seg(C-1)

### 4.3 重要參數
- `Kprime = lenWithCrc(C, A)`（與 UE pipeline 一致）
- out_len：
  - BG1：`68 * Zc`
  - BG2：`52 * Zc`
- bit extraction offset：
  - `bit_off = 2Z`

### 4.4 產出檔案
- `decoded_bits.bin`（A bits，1 bit -> 1 byte）
- `cb_payload_seg%02d.bin`
- `tb_payload_bytes.bin`
- debug：`bitout_seg00.bin`, `bitout_seg01.bin`

## 5) 之後怎麼比對（流程固定化）
在某個 bundle 目錄內（重要：要在該目錄跑，輸出檔才會跟 bundle 放一起方便比）：

```bash
# 1) 跑離線 decoder（只吃 rm_exact）
~/openairinterface5g/cmake_targets/ran_build/build/ldpctest_spx \
  rm_exact_seg00_i8.bin \
  ldpc_cfg.txt \
  decoded_bits.bin

# 2) CB payload byte-by-byte 比對（最關鍵）
cmp -l cb_payload_seg00.bin ue_c_seg00.bin | head
cmp -l cb_payload_seg01.bin ue_c_seg01.bin | head

# 3) TB bytes 比對
cmp -l tb_payload_bytes.bin ue_tb.bin | head
```

### 5.1 判讀準則
若 `cb_payload_segXX.bin` 完全等於 `ue_c_segXX.bin`，且 `tb_payload_bytes.bin` 完全等於 `ue_tb.bin`：

- rm_exact dump 正確 ✅
- ldpctest_spx decoder 正確 ✅
- 之後要查的就不是 decoder，而是：
  - LLR 來源（SpikingRx or demapper）
  - 量化/縮放/符號
  - bundler 對齊 key/regex
  - 或更上游的 demod/chanest/equalization

若 CB/TB 不一致：

優先查：
- rm_exact 是否對應到同一筆 record（frame/slot/idx/tb/rv/seg）
- ldpc_cfg.txt 是否真的是該筆 record 的參數（BG/Zc/A/C/K/F/rv/R）
- `Kprime=lenWithCrc(C,A)` 是否一致
- offset=2Z 是否一致

---

## 6) 這套設計的「最重要核心」
- rm_exact 檔名一定要有 `tb%02u`（防撞檔覆蓋，避免 thread pool 造成幽靈問題）
- ldpctest_spx 只做 decoder，不做 unmatching（縮小責任邊界，讓 cmp 結果可直接定位錯在哪一層）
- 最終驗證只看兩個 oracle：
  - `ue_c_segXX.bin`
  - `ue_tb.bin`
