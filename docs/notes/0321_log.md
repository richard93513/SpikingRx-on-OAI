## 2. 第一個嘗試：把 `y_eq` 改成不 normalize

我一開始的判斷是：

`y_ch` 和 `y_eq` 都做 per-sample normalize，  
可能讓 llr head 變成在學：

```text
(normalized H, normalized H*Y) → raw LLR
```

這種 mapping 很不自然，  
因為 LLR 的 magnitude 本來就跟 symbol 的尺度 / 信心有關。

所以第一個單變因實驗是：

### 修改內容

改 `src/data/dataset_oai_bundle.py`

把 dataset 改成支援：

- `aux_normalize_eq`
- `aux_normalize_ch`

讓 `eq` 和 `ch` 可以分開決定要不要 normalize。

改 `src/train/train_spikingrx_oai.py`

這一版先設定成：

- `y_ch`：normalize
- `y_eq`：不 normalize
- 其他先不動

### 當時跑的訓練指令

```bash
cd ~/SpikingRx-on-OAI && source .venv/bin/activate && PYTHONPATH=src python3 src/train/train_spikingrx_oai.py
```

### 跑出來的重要 log 現象

sample 一開始就看到：

```text
y_eq(masked).std ≈ 182.3
y_ch(masked).std ≈ 1.0
```

也就是 `eq` 直接變成 raw 大尺度。

之後 phase2 一打開：

- `eq loss` 直接炸到 34043
- `val sign` 還是卡在 0.509 ~ 0.510
- `llr_pred std` 雖然從很小變到 1.8~2.0
- 但還是遠小於 `y_llr std ≈ 11.5`

### 第一個實驗的結論

### 結論

把 `y_eq` 改 raw 並沒有解掉主因。

### 理由

- raw eq 的尺度太大，直接把 phase2 的 loss 主導掉
- 雖然模型被迫去追 raw-eq
- 但 `val sign` 沒有真的改善

所以真正的 bottleneck **不是單純 normalize 問題**。

## 3. 第二個關鍵判斷：真正錯的是舊版 llr head 架構

從上一版失敗後，我進一步判斷：

### 舊版 llr head 的問題

它原本的做法本質上是：

```text
[H, W, feature]
→
pool / flatten / FC
→
14400
```

也就是：

先把整張圖壓縮，  
再讓 FC 直接吐 14400 個 bits 的 LLR。

這種做法會逼模型同時學兩件事：

### 1. 該學的

每個位置的 symbol 對應的 bit LLR 是多少

### 2. 不該浪費容量學的

- 哪些位置是 data RE
- 每個 data RE 對應 2 個 bit
- 14400 的排列順序是什麼
- 空間位置如何映射成 bit index

這第二件事其實我們早就知道，不應該交給模型自己瞎學。

## 4. 最關鍵的修改：把 llr head 改成 dense per-RE 2-bit LLR map

這一步是整個聊天室最重要的改動。

### 修改內容

改 `src/models/spikingrx_model.py`

把 llr head 從「整張圖 flatten 成 14400」改成：

先輸出一張 dense LLR map：

```text
[B, 2, H, W]
```

其中：

- 第 0 channel = 該 RE 的第 1 個 bit LLR
- 第 1 channel = 該 RE 的第 2 個 bit LLR

再用 `data_mask` 把有效 data RE 抽出來，  
照固定順序攤平成 `[B, 14400]`

也就是：

```text
(feat, ch_pred, eq_pred)
→
llr_map [2, H, W]
→
gather by data_mask
→
14400
```

同時改 `src/train/train_spikingrx_oai.py`

這版訓練恢復成：

- `y_eq normalize = True`
- `y_ch normalize = True`

也就是回到穩定的 supervision setting，  
但搭配新的 dense llr head。

另外保留：

- hybrid llr loss
- phase schedule
- `llr_temperature = 2.0`

### 這版跑的訓練指令

```bash
cd ~/SpikingRx-on-OAI && source .venv/bin/activate && PYTHONPATH=src python3 src/train/train_spikingrx_oai.py
```

### 結論

## 14. 目前最合理的推論

這 5 個特例 bundle 的失敗型態：

- BER 很穩地接近 `0.48x`
- 不是 decode 小幅劣化
- 比較像是 bit sign 在很多位置上接近一半對一半錯

這通常意味著：

- 很可能不是單純 llr amplitude 太小
- 而是：
  - sign mapping 在那幾個樣本上失真
  - 或那幾個 bundle 的輸入分佈與訓練集中後段大多數 bundle 有明顯差異
  - 也可能 capture 開頭有特殊 transient / 對齊差異 / fullgrid 統計特別不同
