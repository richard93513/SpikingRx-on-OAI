

這一段要由新的 `rmunmatch_spx` 完成。

## 1. rmunmatch_spx 的目的

`rmunmatch_spx.c` 的目的不是自己重寫 unmatching 演算法，而是：

- `rm_exact_spx_seg00_i8.bin`

並用這些輸出去比對 UE oracle：

- `rm_exact_seg00_i8.bin`

## 3. rmunmatch_spx 使用的 OAI 原生 function

這次不是自己重寫 unmatching，而是直接拉 OAI 裡的原生 function。

### 已確認使用的 OAI function

#### 1) deinterleaving

從 header 確認到 prototype：

```c
void nr_deinterleaving_ldpc(uint32_t E, uint8_t Qm, int16_t *e, int16_t *f);
```

也就是 UE pipeline 裡的呼叫方式是：

```c
nr_deinterleaving_ldpc(E, Qm, harq_e, ulsch_llr);
```

#### 2) rate unmatching

從 header 確認到 prototype：

```c
int nr_rate_matching_ldpc_rx(uint32_t Tbslbrm,
                             uint8_t BG,
                             uint16_t Z,
                             int16_t *d,
                             int16_t *soft_input,
                             uint8_t C,
                             uint8_t rvidx,
                             uint8_t clear,
                             uint32_t E,
                             uint32_t F,
                             uint32_t Foffset);
```

注意：這個 branch 沒有 `Ilbrm` 參數。

#### 3) segment E 計算

從 UE path 確認：

```c
segment_parameters->E = nr_get_E(TB_parameters->G,
                                 TB_parameters->C,
                                 TB_parameters->Qm,
                                 TB_parameters->nb_layers,
                                 r);
```

再從 `nr_tbs_tools.c` 確認 `nr_get_E()` 真正公式：

```c
uint32_t nr_get_E(uint32_t G, uint8_t C, uint8_t Qm, uint8_t Nl, uint8_t r)
{
  uint32_t E;
  uint8_t Cprime = C;

  if (r <= Cprime - ((G / (Nl * Qm)) % Cprime) - 1)
    E = Nl * Qm * (G / (Nl * Qm * Cprime));
  else
    E = Nl * Qm * ((G / (Nl * Qm * Cprime)) + 1);

  return E;
}
```

所以 E 不能亂猜，也不能簡單固定 `G/C` 而不經公式。

## 4. 已確認的 UE 真實流程

從：

`openair1/PHY/CODING/nrLDPC_coding/nrLDPC_coding_segment/nrLDPC_coding_segment_decoder.c`

確認到 UE decoder 前半流程是：

```c
int16_t harq_e[E];
nr_deinterleaving_ldpc(E, Qm, harq_e, ulsch_llr);

nr_rate_matching_ldpc_rx(rdata->tbslbrm,
                         p_decoderParms->BG,
                         p_decoderParms->Z,
                         rdata->d,
                         harq_e,
                         rdata->C,
                         rv_index,
                         *rdata->d_to_be_cleared,
                         E,
                         rdata->F,
                         K - rdata->F - 2 * (p_decoderParms->Z));
```

然後接著 build decoder input：

```c
memset(z, 0, 2 * Z * sizeof(*z));
memset(z + Kprime, 127, F * sizeof(*z));
memcpy(z + 2 * Z, d, (Kprime - 2 * Z) * sizeof(*z));
memcpy(z + K, d + (K - 2 * Z), (Kc * Z - K) * sizeof(*z));
```

再做 int16 -> int8 saturate，得到實際餵給 `LDPCdecoder()` 的 `l[]`。

## 5. rmunmatch_spx 開發過程中做過的主要檢查

### (A) 先確認 demapper dump 的 dtype

從 bundle 檔案大小判斷：

- `G = 14400`
- `demapper_llr_f32.bin = 57.6 kB`

因為：

- int8 會是 14400 bytes
- int16 會是 28800 bytes
- float32 會是 57600 bytes

所以確認：

`demapper_llr_f32.bin` 是 float32

### (B) 確認 nr_rate_matching_ldpc_rx 真實 prototype

一開始因為 branch 差異，曾懷疑是否有 `Ilbrm` 參數。

後來實際 grep：

```bash
grep -n "nr_rate_matching_ldpc_rx" \
~/openairinterface5g/openair1/PHY/CODING/nrLDPC_coding/nrLDPC_coding_segment/nr_rate_matching.h
```

確認這個 branch 的 prototype 沒有 `Ilbrm`。

所以最終結論：

```text
SPX_RM_RX_HAS_ILBRM = 0
```

不能再猜有 `Ilbrm`。

### (C) 確認 nr_deinterleaving_ldpc 真實 prototype

實際 grep header 後確認：

```c
void nr_deinterleaving_ldpc(uint32_t E, uint8_t Qm, int16_t *e, int16_t *f);
```

也就是正確呼叫是：

```c
nr_deinterleaving_ldpc(E, Qm, harq_e, soft_in);
```

而不是其他參數順序。

### (D) 確認 UE 端的 E 是哪裡來的

不是自己手算亂猜，而是從 UE path 確認：

```c
segment_parameters->E = nr_get_E(...)
```

所以 `rmunmatch_spx` 裡也要用相同公式。

### (E) 確認 Foffset 的正確值

從 UE 真正呼叫確認：

```text
Foffset = K - F - 2*Z
```

不是 cumulative E，也不是其他 offset。

這點後來修正後，pipeline 明顯更接近 UE。

### (F) 確認 clear 的行為

UE path 會傳入：

```c
*d_to_be_cleared
```

而 offline 單次測試中，為避免 HARQ combine 狀態殘留，`rmunmatch_spx` 採用：

```text
clear = 1
```

也就是每段都視為 one-shot fresh buffer。

目前這樣做可以正常產生輸出，不像一開始那種全 0 / 大段 0。

## 6. 已經排除掉的問題

### 6.1 不是 CMake / 編譯問題

因為：

- `rmunmatch_spx` 已經成功 build
- binary 可以正常執行
- 輸出檔案長度正確
- OAI function 也確實被呼叫

所以這不是「沒編到新版本」或「連結錯 library」造成的主要 mismatch。

### 6.2 不是 function prototype 搞錯造成的全亂

早期有試過錯誤 prototype / 錯誤呼叫方式時，結果會出現：

- 大段都是 0
- 或整段幾乎全 0

現在已經不再是那種 pattern。

目前的輸出有合理值：

- ±11 / ±12

代表：

- function 被正確呼叫
- data 流程大致合理
- 不再是 ABI 爆掉那種錯誤

### 6.3 不是 E 完全算錯

一開始曾試過簡單 `G/C` 或錯的 E 概念，結果也會導致大量 0 / 嚴重錯位。

現在改成 OAI 的 `nr_get_E()` 公式後，輸出已明顯更合理，且 mismatch pattern 穩定。

所以現在不是「E 完全算錯」那種大錯。

### 6.4 不是 Foffset 完全算錯

一開始用錯 Foffset 思路時，也會造成更混亂的結果。

現在改成：

```text
Foffset = K - F - 2*Z
```

之後輸出 pattern 更接近 UE。

所以目前剩下的不是大幅 offset 錯位，而是較細的 sign / convention / exact reconstruction 問題。

### 6.5 不是 scale 大小亂掉

做過幾次 scale 測試：

- `--llr-scale 8`
- `--llr-scale 64`
- `--llr-scale 1`
- `--llr-scale -1`

結果顯示：

- 64 會飽和到 ±127 / ±128
- 1 時大小最接近 UE

在 `scale=1` 時，SPX 與 UE 的絕對值完全一致（至少在大部分 mismatch 區）

這代表：

- LLR magnitude 已經對了

所以現在不是 scale 大小問題，而是 sign / convention / build `l[]` 細節。

## 7. 目前最新的 mismatch 特徵（最重要）

這是目前最關鍵的診斷結果。

### 7.1 cmp 顯示第一個差異位置

- seg00：第 449 byte
- seg01：第 450 byte

也就是說：

前面的 `head_2Z` 基本正確  
問題是從 data 區開始

### 7.2 用 Python 比對後，發現

差異數量：

- seg00: 3642
- seg01: 3599

差異的性質：

之前分析出：

- `same_abs_on_diff = 100%`
- `opp_sign_on_diff = 100%`

也就是：

在 mismatch 的那些點上：

```text
|spx| = |ue|
sign(spx) = -sign(ue)
```

換句話說：

- 值大小對了
- 符號相反

### 7.3 依區段分析後得到

完全正確的區段：

- `head_2Z`（1 ~ 448）
- `filler`（4777 ~ 4928）在某些版本下固定對

有問題的區段：

- `data_before_filler`（449 ~ 4776）
- `tail_after_filler`（4929 ~ 15232）

而且這些錯誤都是：

```text
only sign flipped
```

### 7.4 非純 sign 錯誤的部份

後來又做了更細的分析，發現非 sign mismatch 的區域只有：

- `4777 ~ 4928`

也就是剛好是 filler bits 區段（長度 152 = F）。

曾經出現：

- `spx = -128`
- `ue  = +127`

這表示：

- filler bits 區不能跟一般 data 一樣翻號
- filler 區必須固定為 `+127`

這個問題後來已經辨識出來。

目前程式的設計方向已經確定是：

## 9. 目前最可能還沒完全對齊的地方

目前已經非常明確，下一步要集中檢查的重點有兩個：

### 9.1 d_unmatched 與 l[] 之間的 sign convention 到底在哪裡翻

目前最大的未解點是：

到底是：

- demapper 輸出要先翻號？
- 還是 d_unmatched -> l[] 重建時某些區段要翻號？
- 還是 UE dump 的 rm_exact 本身就使用了某種特定 sign convention？

目前已知：

- mismatch 的絕對值對了
- data 區只差符號
- filler / head 區不應該翻號

所以最像是：

- data 區的 sign convention 還沒完全跟 UE 對上

但翻號應該放在：

- quantize 前後？
- deinterleave 後？
- unmatch 後？
- build `l[]` 時？

這個還需要進一步用更細的 dump 來確認。

### 9.2 rm_exact_segXX_i8.bin 這個 UE dump 到底是在 UE pipeline 的哪一個精確 stage dump 出來的

目前我們一直把它視為：

```text
"fed into LDPCdecoder() 的 exact int8 buffer"
```

但下一步要再確認：

- 是不是 dump 在 `simde_mm_packs_epi16()` 之後？
- 還是 dump 在某個額外 sign-conversion 後？
- 是否還有 check_crc / SIMD pack / packing convention 的影響？

這點非常重要，因為現在整條 pipeline 都很像是：

- 我們 reconstruct 的 `l[]` 跟 UE `l[]` 的內容只有 sign 規約差異

所以 dump 點如果再精確確認，會更快收斂。

## 10. 下一步建議怎麼檢查與修改

下面是最建議的下一步順序。

### 步驟 1：先在 UE path 補 dump d[]（rate-unmatched int16）而不是只 dump 最終 l[]

目前我們只有：

- `demapper_llr_f32.bin`
- `rm_exact_segXX_i8.bin`

但中間最重要的 `d_unmatched` 還沒有 oracle。

建議在 UE 的：

`nrLDPC_coding_segment_decoder.c`

中，在 `nr_rate_matching_ldpc_rx(...)` 之後、build `z[]/l[]` 之前，加 dump：

- `ue_d_unmatched_seg00_i16.bin`
- `ue_d_unmatched_seg01_i16.bin`

這樣就能直接比較：

- `SPX d_unmatched vs UE d_unmatched`

一旦這個對上，就可以把問題切成兩半：

- `demapper -> deinterleave -> unmatch` 是否正確
- `d_unmatched -> z[] -> l[]` 是否正確

這會比現在只拿最終 rm_exact 在黑盒子裡猜 sign 快很多。

### 步驟 2：若 d_unmatched 已經跟 UE 只差 sign，就能直接定位 sign flip stage

到時候可以判斷：

- 若 `d_unmatched_spx == d_unmatched_ue`  
  問題在 build `l[]`

- 若 `d_unmatched_spx == -d_unmatched_ue`  
  問題在 demapper sign / deinterleave / unmatch 前半段

這是現在最需要的診斷。

### 步驟 3：確認 rm_exact_segXX_i8.bin dump 點與 SIMD pack 前後的關係

UE code 中有：

```c
pl[j] = simde_mm_packs_epi16(pv[i], pv[i + 1]);
```

若要完全 bit-exact，理論上最好確認：

- UE dump 是在 pack 前還是 pack 後
- 是不是 exact `l[]` memory dump
- 是否與我們目前的逐點 `clamp_i8()` 完全等價

雖然目前大小看起來對，仍建議確認。

### 步驟 4：暫時不要再亂改 CMake、prototype、E/Foffset

這些大方向已經檢查很多次，現在重點不是這些。

目前更應專注在：

- sign convention
- exact stage dump
