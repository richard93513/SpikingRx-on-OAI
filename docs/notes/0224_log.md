## 10. 最終本聊天室的核心結論（目前確定到哪）

### 已經確定的事
✅ UE TB bits == gNB TX bits（errors=0）  
=> OAI pipeline 正常，通道/連線不是根因。

✅ 外部 ldpctest_spx decode 完雖然可收斂（n_iter=2），但 output TB/CB 與 UE 不一致  
=> 外部 wrapper 仍未 100% 複製 UE pipeline，且 mismatch 導致 BER≈0.5。

✅ 你目前最致命的工程問題是：檔案命名/idx 不一致  
- ue_c/ue_tb/ldpc/txbits 都帶 idx（例如 idx055501）  
- ue_rm_exact 卻不帶 idx，只用 seg_counter（1797/1357/…）  
=> 你永遠很難保證拿到「同一筆」對應的 seg00/seg01 來 concat。

### 還沒完全解開的事（下一步要做）
若外部 ldpctest_spx 直接吃「UE LDPCdecoder 真正的 int8 input buffer（ue_rm_exact）」仍 mismatch：
- 就不是 rate-match/deinterleave 問題，而是：
  - 你餵的不是同一筆（idx 對不上）
  - 或 ldpctest_spx 解碼參數/介面仍不等同 UE（例如 outMode、Kprime、R、Ncb 等）
  - 或 decoder library/模式選到不同版本（CPU/GPU/不同 LDPC impl）

## 11. 下一步行動：統一 idx（你提供的 dlsch_coding idx 寫法）

你目前在 nr_dlsch_coding.c 的 idx 寫法是：

```c
static inline uint32_t spx_make_idx(int frame, int slot)
{
  return (uint32_t)frame * 100u + (uint32_t)slot;
}
const uint32_t this_idx = spx_make_idx(frame, slot);
```

並用在檔名：
- `f%04d_s%02d_ldpc_idx%06u_...`

問題：segment decoder 層級沒有 frame/slot，所以不能直接重算 idx。

### 正確方向（本聊天室結論）
要做「全域 idx」：由上層（有 frame/slot 的地方）決定一次，寫到全域變數，讓底層（segment decoder）也能用同一個 idx 命名。

這樣 ue_rm_exact 檔名就能變成跟其他 dump 一樣：
- `...ue_rm_exact_idx%06u_seg00...`
- `...ue_rm_exact_idx%06u_seg01...`

同一個 idx 下，seg counter 要「每筆 TB reset」成 0/1（而不是一路累加到 1797）。
