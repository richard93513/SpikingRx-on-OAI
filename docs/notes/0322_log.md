## 2. 第一輪檢查：先看資料統計與 cfg

### 2.1 一開始誤把 fullgrid 當 float32 讀，得到錯誤結論

你先用 `np.fromfile(..., np.float32)` 讀 `fullgrid.bin`，結果出現：

- `mean=nan`
- `std=nan`
- `reshape` 失敗，大小對不上 `(4,14,1272)`

例如：

- `fullgrid count = 28676`
- 但正確應該是 `4*14*1272 = 71232`

一開始我誤判成 dump 格式有問題，但後來你又補做了檔案大小檢查，發現：

```bash
python3 - <<'PY'
from pathlib import Path

p = Path('~/SpikingRx-on-OAI/spx_records/bundle/f0130_s01_idx013001_rnti1234_harq01_rv2/fullgrid.bin').expanduser()
data = p.read_bytes()
print("bytes =", len(data))
print("float32 count =", len(data)//4)
print("float16 count =", len(data)//2)
print("int16 count   =", len(data)//2)
print("int8 count    =", len(data))
PY
```

結果：

```text
bytes = 114704
```

這個值其實完全合理，因為：

- header = `uint16[8] = 16 bytes`
- payload = `14 * 2048` 個 complex sample
- 每個 complex sample = I,Q 各一個 `int16 = 4 bytes`
- payload = `14 * 2048 * 4 = 114688`
- 總和 = `114688 + 16 = 114704`

所以後來修正結論：

- `fullgrid.bin` 不是壞掉，而是它的格式本來就不是 `float32 raw tensor`，而是 `header + int16 IQ payload`。

### 2.2 正確解讀 fullgrid header 與 payload

你後來用正確方式讀 `fullgrid.bin`：

- `raw[:8].view(np.uint16)` 當 header
- `raw[8:]` 當 int16 IQ payload

印出幾個壞 bundle 與第一個正常 bundle 的資訊，看到：

- `frame`
- `slot`
- `start_symbol=1`
- `n_sym=14`
- `first_sc=388`
- `used_sc=1272`
- `number_symbols=13`
- `G=14400`

這些 header 都正常，而且壞的 bundle 與正常的 bundle 之間：

- `pdsch_cfg.txt`
- `ldpc_cfg.txt`

也幾乎都相同，除了 `harq_pid / rv_index` 這些本來就會變。

這一步的結論是：

- 問題不是 cfg 配置切換，也不是 dump 檔案格式錯誤。

## 3. 找到真正關鍵：fullgrid amplitude regime 分成兩群

- `min/max` 大約在 `±260 ~ ±264`

- `std` 大約都還是 `114`

### 3.3 掃前 30 個 bundle，發現切換點固定

## 16. 最後分析應該怎麼做

你最後問到：

之後是不是全部跑完後，再做全部 dB 的錯誤率分析？  
是不是用總錯誤 bit 數除以全部 bundle 的總 bit 數？

這個方向是對的。

正式主指標應該是：

```text
group BER = sum(bit_errors) / sum(total_bits)
```

因為目前每個 bundle 的資訊裡有：

- `bit_errors`
- 每包 bit 長度固定 `A = 9480`

所以某個 dB 組的正式 BER 應該算成：

```text
BER_group = 所有有效 bundle 的 bit_errors 總和 / (有效 bundle 數 × 9480)
```

而不是只看：

- `nonzero bundle count`

`nonzero bundle count` 只能當輔助指標，不能取代真正 bit-level BER。
