## 2. 一開始原本懷疑：是不是 input 被壓太兇

當時發現目前 loader 會把原始 OAI fullgrid 變成：

```text
原始有效區域: 14 × 1272
↓
頻域 mean pooling
↓
32 × 64
↓
再複製成 T=5
↓
dataset 又只取第一個 time step
↓
實際送進模型的是 [1, 2, 32, 64]
```

### 原始 loader 的關鍵問題

在 `src/data/oai_to_spikingrx_tensor.py` 裡：

先從 header 取出：

- `n_sym = 14`
- `used_sc = 1272`

對 `1272 → 64` 做區間平均：

```python
freq_out[:, i] = grid_used[:, s:e].mean(axis=1)
```

再把 `14 → 32` 用 zero padding 補高

再把同一張 `chw` 複製成 T 個 time step

### 我們當時的懷疑

這種做法很可能：

- 破壞 constellation / 頻域細節

尤其是：

```text
1272 → 64
```

這個壓縮比太激進。

## 5. 改過的 4 個程式內容與理由

### A. `src/data/oai_to_spikingrx_tensor.py`

#### 原本

輸出固定做成 `32 × 64`

- `1272 → 64` 用 mean pooling
- `14 → 32` 用 zero padding
- 再 fake 成 `T=5`

#### 修改後

新增 `keep_rect=True` 模式，保留原始有效 BWP：

```text
[1, T, 2, 14, 1272]
```

也就是不再先做：

```text
1272 → 64 mean pooling
```

#### 修改理由

要先驗證：

是不是 input encoding 壓太多，才導致學不起來。

---

### B. `src/data/dataset_oai_bundle.py`

#### 原本

讀 `demapper_llr_f32.bin`

用：

```python
load_oai_fullgrid(..., H_out=32, W_out=64, T=5)
```

`squeeze(0)` 後只取 `x[:1]`

實際輸出 shape 是：

```text
[1, 2, 32, 64]
```

#### 修改後

改成使用：

```python
load_oai_fullgrid(..., T=1, keep_rect=True)
```

輸出改成：

```text
[1, 2, 14, 1272]
```

保留 per-sample z-score normalization

#### 修改理由

讓模型真正看到原始 rectangular fullgrid，而不是被先壓成 `32×64` 的版本。

---

### C. `src/models/spikingrx_model.py`

#### 原本

- base channel 較大
- readout 在 CPU
- readout 使用：

```text
AdaptiveAvgPool2d(4,8)
→ FC → 14400
```

#### 修改後

改成較輕量版本，避免 GTX 950M 壓力過大

- base channel 改小（例如 8）
- readout 留在 GPU

仍然保留：

```text
AdaptiveAvgPool2d → FC → 14400
```

但 pool 尺寸改成較適合 `14×1272` 的設定，例如：

```text
(7, 24)
```

#### 修改理由

先確保在 rectangular input 下可以順利訓練，並避免 GPU 記憶體 / 速度問題。

---

### D. `src/train/train_spikingrx_oai.py`

這支改了兩個階段。

#### 第 1 階段：raw LLR regression 版

- dataset 改成吃 rectangular input
- loss 改成直接學 raw LLR

使用：

```python
loss_fn = nn.SmoothL1Loss(beta=1.0)
```

#### 第 2 階段：sign-only BCE 版

後來為了確認是不是「raw regression 太難」，又改成：

```python
target_sign = 1 if LLR >= 0 else 0
loss = BCEWithLogitsLoss
```

也就是把任務改成：

```text
fullgrid → LLR sign classification
```

#### 修改理由

這是一個很重要的判別實驗：

- 如果 raw LLR regression 學不起來，但 sign classification 學得起來  
  → 表示 input 其實有資訊，只是 regression 太難

- 如果連 sign classification 都學不起來  
  → 表示問題更偏向 input / task 本身

## 7. 我們針對論文做了什麼查核

這一段很重要，因為中間有幾次推論，最後已經重新修正成較嚴謹的版本。

### 論文查核重點 1：SpikingRx 的輸入到底是不是 full OFDM grid

查到論文原文寫的是：

SpikingRx 取代：

- channel estimation
- equalization
- interpolation
- symbol demapping 等 block

輸入是：

```text
whole 5G-NR resource grid in a slot
```

而且 explicitly 提到：

```text
different DMRS configurations as input
```

輸出是可用作 LLR 的 soft outputs。

#### 結論

這支持以下判斷：

論文輸入不可能是 equalized symbols

因為如果輸入已經 equalized，就不可能再宣稱「取代 channel estimation / equalization」。

所以這件事已經確認：

- 你現在用 fullgrid 當 input 的大方向沒有錯

---

### 論文查核重點 2：論文有沒有明確說「額外輸入 pilot mask / DMRS mask」

這裡要很嚴謹：

#### 查核結果

我沒有在 2024 那篇 SpikingRx 原文裡看到明確寫：

```text
input = received grid + pilot mask
```

它比較像是寫：

- whole OFDM grid
- 以及 different DMRS configurations as input

但沒有明講是額外加一個 mask channel。

#### 結論

所以不能把「額外 DMRS mask」當作 2024 SpikingRx 的既定設定。

---

### 論文查核重點 3：相關文獻是不是有顯式提供 pilot 資訊

這部分查到兩個方向：

#### (a) DeepRx

DeepRx 是 fully convolutional deep learning receiver，屬於 neural receiver 的代表工作之一。這類工作通常會特別處理 pilot / data 的輸入結構。

#### (b) NeuromorphicRx（2025，同方向後續工作）

作者後續的 2025 NeuromorphicRx 論文，已經更明確把 pilot 相關資訊顯式放進輸入。

#### 結論

這代表：

即使 2024 SpikingRx 原文沒有明講 pilot mask，  
相關 neural receiver 文獻與後續工作，  
確實有把 pilot 資訊顯式提供給網路的做法。
