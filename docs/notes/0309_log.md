## 1. 一開始的目標與核心原則

這次工作的核心目標不是單純讓程式「能跑」，而是：

讓 offline 的 `rmunmatch_spx`  
真正重建出 UE LDPC decoder 看到的 exact input

也就是：

```text
SpikingRx / offline pipeline
==
UE decoder pipeline
```

### 核心原則

這次 debugging 全程都遵守一個非常重要的原則：

**不要自己重寫 PHY algorithm**  
要直接拉 OAI 裡面原本的 function 來用

具體來說，這次要對齊的關鍵 OAI function 是：

- `nr_dlsch_unscrambling()`
- `nr_deinterleaving_ldpc()`
- `nr_rate_matching_ldpc_rx()`

這個原則非常重要，因為：

- 3GPP / OAI 的 bit-level 行為很容易有細節差
- 自己重寫很容易 sign、offset、padding、pack path 不同
- 要做的是 **exact reproduction**，不是 approximate reproduction

## 2. 問題起點：懷疑 rmunmatch_spx 少了 UE pipeline 的某一步

在這次聊天之前，已經有一個很明確的懷疑：

`rmunmatch_spx` 雖然有做 deinterleave + rate matching rx + exact pack，  
但輸出仍然跟 UE 的 `rm_exact` dump 對不起來

這時候懷疑的方向是：

- 是否少了 unscrambling
- 是否少了 deinterleaving
- 是否某些參數來源不對
- 是否前處理與 UE pipeline 不一致

這時候不是直接亂改，而是先用 log + source trace 找 UE 真正的 call chain。

## 9. 改寫 `rmunmatch_spx.c` 的目標

原本 `rmunmatch_spx` 大致是：

```text
demapper_llr_f32
→ int16
→ deinterleaving
→ rate_matching_rx
→ UE exact z rebuild
→ simde_mm_packs_epi16
```

新版要改成：

```text
demapper_llr_f32
→ int16
→ nr_dlsch_unscrambling()
→ deinterleaving
→ rate_matching_rx
→ UE exact z rebuild
→ simde_mm_packs_epi16
```

### 新版 CLI

改成：

```bash
./rmunmatch_spx <demapper_llr_f32.bin> <ldpc_cfg.txt> <pdsch_cfg.txt> <rm_exact_spx_prefix> [--llr-scale <float>]
```

### 新增 `pdsch_cfg.txt` 解析

新增 `pdsch_cfg_t` 與 `load_pdsch_cfg_kv()`，讀出：

- rnti
- G
- dlDataScramblingId
- physCellId

### 新增 OAI unscrambling call

在全長 G 的 LLR 上先做：

- `nr_dlsch_unscrambling(...)`

然後再 per-segment 做：

- `nr_deinterleaving_ldpc()`
- `nr_rate_matching_ldpc_rx()`
- exact UE z rebuild
- exact pack

## 11. 第二次 build 問題：`undefined reference to get_nrUE_params`

### 問題現象

link `PHY_NR_UE` 之後，又出現：

```text
undefined reference to `get_nrUE_params'
```

### 原因

`nr_dlsch_decoding.c.o` 裡不只有 `nr_dlsch_unscrambling()`，還有 `nr_dlsch_decoding()`。  
後者會 reference `get_nrUE_params()`。

但 `rmunmatch_spx` 根本不需要跑 `nr_dlsch_decoding()`，只是 linker 看到同一個 object 內有這個 symbol 依賴。

### 檢查方法

用指令查 prototype：

```bash
grep -Rsn "get_nrUE_params" . | head -20
```

查到：

```c
nrUE_params_t *get_nrUE_params(void)
```

### 修法

在 `rmunmatch_spx.c` 裡補一個 weak stub：

```c
__attribute__((weak))
nrUE_params_t *get_nrUE_params(void)
{
  static nrUE_params_t p;
  memset(&p, 0, sizeof(p));
  return &p;
}
```

這只是為了滿足 linker，不是要真的跑 UE decoder。

## 15. 找出 `post_unscram` 全 0 的真正根因：`byte2m128i[]` 沒初始化

### 檢查 `byte2m128i`

用指令：

```bash
grep -Rsn "byte2m128i" openair1/PHY/NR_TRANSPORT openair1 | head -20
nl -ba openair1/PHY/NR_TRANSPORT/nr_scrambling.c | sed -n '1,58p'
```

查到：

`byte2m128i` 定義在：
- `openair1/PHY/nr_phy_common/src/nr_phy_common.c`

初始化函式是：
- `void init_byte2m128i(void)`

### 根因判斷

standalone `rmunmatch_spx` 沒經過完整 UE softmodem 初始化流程，  
所以：

```text
byte2m128i[] 沒初始化
→ nr_codeword_unscrambling() 用到全 0 LUT
→ 乘下去整包 LLR 變 0
```

這完全解釋了前面看到的：

- `post_unscram_i16 = all zeros`

---

## 16. 最終修法：在 `rmunmatch_spx.c` 補 `init_byte2m128i()`

### 新增 declaration

在 `rmunmatch_spx.c` 裡新增：

```c
extern void init_byte2m128i(void);
```

### 在 `main()` 裡呼叫

在：

```c
logInit();
```

後面立刻加：

```c
init_byte2m128i();
```

也就是：

```c
logInit();
init_byte2m128i();
```

這就是讓 `nr_codeword_unscrambling()` 在 x86 SIMD 路徑下能正常工作的最後關鍵。
