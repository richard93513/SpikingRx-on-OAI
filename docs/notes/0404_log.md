然後要把原本的：

## 1. 一開始的問題：不知道怎麼得到正確 SNR

一開始曾經試過很多方法，想從 UE 端或 dump 出來的 receiver-side 資料反推 SNR，但最後確認這些路都不可靠。

試過但最後判定不適合的東西

nvar  
一開始 dump 了 nvar  
發現數值有乘 1024 的問題  
後來確認 OAI 本來就有 scale  
最終判定：nvar 本身不能穩定反映真實 channel SNR  

|H|^2  
想用 channel estimate power 當 signal power  
但結果幾乎被通道本身主導，不隨 noise 明顯改變  

DMRS residual  
想用 |y - Hx|^2 / |Hx|^2  
結果不同 noise 設定差異不夠穩定  

fullgrid / rxdataFext energy  
看 signal energy 幾乎固定  
noise 被 signal 淹掉  

eqsymb  
因為 equalization / normalization 後，已經不是 raw channel noise 狀態  

LLR mean(|LLR|)  
LLR 本身有 clipping / saturation 問題  
不適合當物理 SNR  

UE measurement SNR  
類似 rx_power_avg_dB - n0_power_avg_dB  
這是平均後、量化後的 measurement  
解析度不足，無法乾淨區分不同 AWGN 設定  

---

## 2. 最後確認的正確方向：SNR 一定要在加 noise 的地方算

最後確定真正正確的想法是：

不要在 UE 端猜 SNR  
要在 channelmod / rfsim 的 add_noise 當下，直接量 signal power 與 noise power  

也就是：

SNR = E[|signal|²] / E[|noise|²]

這才是這個實驗真正要的 channel-side ground-truth SNR。

---

## 3. 精準定位到真正加 noise 的程式位置

做了 grep 後，定位到真正的加噪點在：

~/openairinterface5g/radio/rfsimulator/apply_channelmod.c

關鍵搜尋結果顯示：

noise_per_sample 在這裡計算  
gaussZiggurat() 在這裡實際加到 I/Q 上  

所以這裡就是唯一正確的 SNR 計算點  

也就是這幾行的附近：

out_ptr->r += rx_tmp.r * pathLossLinear + noise_per_sample * gaussZiggurat(...)  
out_ptr->i += rx_tmp.i * pathLossLinear + noise_per_sample * gaussZiggurat(...)  

---

## 4. 一開始做 SNR 計算時踩過的坑

一開始改 apply_channelmod.c 的時候，踩了幾個很重要的坑：

(1) 直接整個覆蓋原檔，導致 include 壞掉  

曾經因為整檔重寫，少了 OAI 原本的 include，導致：

c16_t  
cf_t  
channel_desc_t  

這些 type 通通找不到，編譯直接爆。

(2) 用 sed patch 太多次，把檔案插爛  

後來一度用 sed 插 patch，但因為反覆改，造成：

snr_done_flag 被插了很多次  
printf 的字串被拆行  
大括號結構壞掉  
scope 錯誤  

最後結論是：

這種核心 C 檔要直接給完整覆蓋版，不要一直 patch  

(3) 一開始算整個 buffer 的 SNR，會出現 signal_power = 0  

這是很重要的 insight。

因為不是每個 sample 都真的有 data-bearing signal，很多 sample 是 idle / 沒有有效訊號。  
如果把整個 nbSamples 全部平均，會出現：

signal_power ≈ 0  
noise_power > 0  
→ SNR = -inf  

所以後來改成：

只在有有效 signal 的 sample 上統計  

這是必要的。

---

## 5. 後來的 SNR 定義：不是取全部平均，而是取 top 10%

後來討論過幾種做法：

不採用的做法  

全部 sample 平均 SNR  
錯，因為會把沒有資料的 sample 一起平均進去  

只取最大 SNR  
可以用，但太容易被問「為什麼只取最大？」  

用某個 signal threshold 近似 top10%  
速度快，但理論上不夠漂亮  
教授一問 threshold 怎麼訂，容易卡住  

最後採用的做法  

對有效 signal sample 的 SNR 做排序，取 top 10% 平均  

這樣的理由是：

避免 idle sample 污染  
比只取 max 穩定  
比全部平均更符合 data-bearing symbol 的 SNR  
以 percentile 定義，理論上比較可 defend  

也就是最後採用的是：

top10% SNR average  

---

## 6. 為了速度，SNR 不是每個 slot 都一直算，而是摘要後只 dump 一次

一開始如果每次都收全部 sample、做 bubble sort，速度超慢。  
後來改成：

最終採用的策略  

只在 rxAnt == 0 時計算  
不要每個 slot 都一直印  
每次 call 收到一批有效 SNR sample  
對該批做 top10% 平均  
收集多個 window  
最後用多個 window 的平均，產生一個最終 SNR  

最後固定的摘要策略  

在 apply_channelmod.c 中：

TARGET_SNR_WINDOWS = 8  
MIN_VALID_SNR_SAMPLES = 2048  

每個 noise setting 收集 8 個有效 window  
每個 window 各自算 top10% SNR  
最後把這 8 個 window-level SNR 做平均  
只 dump 一次  

也就是最後的 SNR 不是「第一個 slot 的值」，而是：

8 個有效 window 的 top10% SNR 平均  

這比只取第一個穩得多。

## 7. 最後的 channel SNR dump 形式

最後在 apply_channelmod.c 裡加了 dump，輸出到：

~/SpikingRx-on-OAI/spx_records/raw/snr_top10_run_summary.txt  

內容格式像這樣：

noise_power_db=-5.000 snr_top10_db=27.276879 used_windows=8  

重點是：

noise_power_db：這次模擬設定的 noise_power_dB  
snr_top10_db：經過 top10% + 8 windows 平均後的最終 SNR  
used_windows=8：表示確實用到 8 個 window

## 8. bundler 也改了：把 SNR summary 放到 bundle 外層

不是把 SNR 塞進每個 bundle 裡，也不是塞進 ldpc_cfg 或 pdsch_cfg。

最後決定的做法  

在 bundler 中：

如果 RAW_DIR 裡有 snr_top10_run_summary.txt  
就把它複製到 BUNDLE_DIR 根目錄  

也就是 snapshot 裡會長成：

bundle_noise_power_minus4.3dB/  
  ├── snr_top10_run_summary.txt  
  ├── batch_gate_summary.json  
  ├── f0132_s11_idx...  
  ├── f0133_s01_idx...  
  └── ...  

而不是每個 bundle 各放一份。

這樣的語意最合理，因為：

一個 noise_power_dB 資料集 → 對應一個 SNR 摘要  

不是每個 bundle 各有自己獨立 SNR。

## 11. 中途有修過的一個顯示 bug：-4.65 被顯示成 -4.7

## 14. 關於中間 cliff 跳躍的判斷

後來有特別確認：

中間跳躍很大，是不是圖有問題？

目前的判斷是：

不一定是錯  
而是因為某個點還有 2~3 個錯 bit，下一個點變成 0 error  

在你現在每個點大約：

1500 bundles × 9480 bits ≈ 1.42e7 bits  

的情況下，只要從「還有少數 bit error」跳到「完全 0 error」，在 semilogy 上就會非常像斷崖。

所以目前這個 cliff 跳躍本身是可以接受的，不一定要硬補更多點。
